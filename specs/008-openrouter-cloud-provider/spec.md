# Feature Specification: OpenRouter Cloud-Model Provider

**Feature Branch**: `008-openrouter-cloud-provider`

**Created**: 2026-06-08

**Status**: Draft

**Input**: User description: "Add an optional OpenRouter cloud-model provider as an alternative to the local Qwen model."

## Overview

Today the post-session debrief's grammar/coherence feedback is produced by a local
Qwen model that must be loaded into memory. Some users do not have enough RAM to run
that model. This feature adds an **opt-in cloud mode** that routes the feedback step
to an OpenRouter-hosted model instead of loading the local Qwen weights, while leaving
the default local/offline experience completely unchanged for everyone who does not opt
in. Cloud mode swaps **only the language-model feedback step** — speech playback and
transcription continue to use their existing local models.

## Clarifications

### Session 2026-06-08

- Q: For first-time cloud-mode entry, how is the privacy trade-off (attempt transcripts sent to OpenRouter) consented to? → A: A one-time informational disclosure shown at first-run token capture, plus a brief reminder on each cloud run; invoking the `--cloud` opt-in is itself the consent. No separate blocking confirmation prompt.
- Q: How is the OpenRouter model id configured? → A: A single `model:` key in a dedicated YAML config file at `~/.speakloop/openrouter.yaml` (default `qwen/qwen3.7-max` when the file or key is absent). No environment-variable override for the model id — the YAML file is the one place.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Run a practice session without the local language model (Priority: P1)

A user whose machine cannot load the local Qwen model opts into cloud mode and completes
a full practice session — listen, attempt, and debrief feedback — where the grammar and
coherence feedback is generated by an OpenRouter-hosted model rather than the local one.
The local Qwen weights are never loaded into memory.

**Why this priority**: This is the core value of the feature — it unblocks users who are
RAM-constrained from getting language feedback at all. Without it, nothing else matters.

**Independent Test**: On a machine (or simulated environment) where the local Qwen model
is absent or cannot load, invoke the cloud variant of the practice command with a valid
token configured and confirm a complete debrief with grammar feedback is produced and no
attempt is made to load the local Qwen weights.

**Acceptance Scenarios**:

1. **Given** a valid OpenRouter token is stored and cloud mode is selected, **When** the user runs a practice session and reaches the debrief, **Then** the grammar/coherence feedback is generated via OpenRouter and the report is written exactly as a local-mode report would be (same structure, same `schema_version`).
2. **Given** cloud mode is selected, **When** the feedback step runs, **Then** the local Qwen model is not loaded into memory at any point during the run.
3. **Given** cloud mode is selected and the local Qwen model is not installed, **When** the user runs a practice session, **Then** the session still completes the debrief without prompting the user to download the local language model.

---

### User Story 2 - One-time token capture, then silent reuse (Priority: P1)

The first time a user runs anything in cloud mode, the system prompts once for their
OpenRouter API token and stores it persistently in the project's existing per-user
configuration location. Every subsequent cloud run reuses the stored token silently and
never prompts again — unless the stored token is missing or rejected by OpenRouter.

**Why this priority**: Cloud mode is unusable without a credential, and a tool that asks
for the token on every run would be unacceptable. Capture-once-then-reuse is part of the
minimum viable cloud experience.

**Independent Test**: Delete any stored credential, run the cloud variant once and provide
a token when prompted, confirm the action completes; then run the cloud variant a second
time and confirm no token prompt appears and the action completes using the stored token.

**Acceptance Scenarios**:

1. **Given** no OpenRouter token is stored, **When** the user runs the cloud variant for the first time, **Then** the system prompts once for the token, stores it persistently, and proceeds.
2. **Given** a previously stored, valid token exists, **When** the user runs the cloud variant again, **Then** no prompt appears and the stored token is used.
3. **Given** a stored token is rejected by OpenRouter as invalid, **When** the user runs the cloud variant, **Then** the system surfaces a clear, actionable error rather than silently reusing the rejected token.

---

### User Story 3 - Local/offline experience is unchanged for non-adopters (Priority: P1)

A user who never opts into cloud mode sees no change whatsoever: the default command runs
the local Qwen flow, makes no network calls after model download, never asks for an
OpenRouter token, and produces byte-for-byte the same report output as before this feature.

**Why this priority**: The constitution guarantees a fully local, offline default. Any
regression to that path — an extra prompt, a network call, a changed report — would be a
non-negotiable failure. This story protects the existing experience.

**Independent Test**: Run the existing local command with no cloud opt-in, with no
OpenRouter token configured and no network connectivity, and confirm the session behaves
identically to the pre-feature behavior, including report content.

**Acceptance Scenarios**:

1. **Given** the user does not opt into cloud mode, **When** they run a practice session, **Then** the local Qwen flow runs unchanged and no OpenRouter token is requested.
2. **Given** the user does not opt into cloud mode and has no network access, **When** they run a practice session, **Then** the session completes fully offline exactly as before.
3. **Given** the user does not opt into cloud mode, **When** the debrief report is written, **Then** its content and `schema_version` are identical to the pre-feature output for the same inputs.

---

### User Story 4 - Change the cloud model with one setting (Priority: P2)

A user swaps the OpenRouter model used for feedback by editing a single, obvious setting
(one config line or one environment variable) and re-running the cloud command. No code
changes are required, and the new model is used on the next run.

**Why this priority**: Model choice is the most likely thing a cloud user will want to tune
(cost, quality, availability). It is high value but not required for a first working slice.

**Independent Test**: Change the designated model-id setting to a different valid OpenRouter
model, re-run the cloud command, and confirm the request is routed to the newly configured
model without any code change.

**Acceptance Scenarios**:

1. **Given** the model-id setting is unset, **When** the user runs the cloud command, **Then** the default model `qwen/qwen3.7-max` is used.
2. **Given** the user changes the single model-id setting to another model, **When** they re-run the cloud command, **Then** the newly configured model is used for the feedback request.
3. **Given** the model-id setting names a model OpenRouter does not recognize, **When** the cloud command runs, **Then** the resulting error is surfaced clearly to the user.

---

### User Story 5 - Tune cloud behavior by editing a dedicated prompt file (Priority: P3)

A user edits a dedicated cloud-model system-prompt file to change how the cloud model
behaves, and the next cloud run reflects the edit. This prompt file is entirely separate
from the prompt used by the local Qwen flow; editing it never affects local mode, and the
local prompt is never read or referenced by cloud mode.

**Why this priority**: Prompt tuning is a power-user refinement. It is valuable for
adjusting feedback tone/depth without code changes, but the feature delivers value without
it.

**Independent Test**: Edit the cloud system-prompt file, run the cloud command, and confirm
the cloud model's output reflects the edited instructions; separately confirm the local
mode output is unaffected by the edit.

**Acceptance Scenarios**:

1. **Given** the cloud system-prompt file exists, **When** the user edits it and re-runs the cloud command, **Then** the cloud model's behavior reflects the edited prompt on the next run, with no code change.
2. **Given** the user edits the cloud system-prompt file, **When** they run a local-mode session, **Then** the local feedback is unchanged because local mode does not read the cloud prompt file.
3. **Given** cloud mode runs, **When** it composes its request, **Then** it uses only the cloud prompt file and never reads or references the local Qwen flow's prompt.

---

### Edge Cases

- **No token stored and prompt declined/empty**: If the user does not supply a token when first prompted (cancels or enters an empty value), cloud mode stops with a clear, actionable message and does not store an empty credential.
- **Stored token rejected (authentication failure)**: A `401`-class rejection from OpenRouter produces an actionable error explaining how to update the stored token or fall back to local mode; the rejected token is not silently reused on the next run without re-surfacing the problem.
- **No network connectivity in cloud mode**: A network failure while contacting OpenRouter surfaces a clear error and does not crash the session; the user is pointed to retry or fall back to local mode.
- **OpenRouter transient/server error or timeout mid-request**: A non-auth failure during the feedback request is surfaced clearly and the session degrades gracefully (the rest of the debrief is preserved, consistent with how local-mode feedback failures are handled today) rather than crashing.
- **Cloud mode selected but speech/transcription models absent**: Cloud mode only replaces the language-feedback step; the local speech and transcription models are still required and their absence is handled exactly as in local mode.
- **Unrecognized or malformed model-id setting**: A model id OpenRouter does not accept yields a clear error identifying the bad model setting.
- **Both a stored token and an environment-provided token present**: A deterministic, documented precedence decides which token is used (see Assumptions).
- **Malformed cloud response**: If the cloud model returns output that cannot be parsed into feedback, cloud mode degrades gracefully the same way local mode does on an unparseable response, without crashing the session.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST provide an explicit, discoverable way for the user to opt into "cloud mode" when running a practice session, distinct from the default local-mode invocation.
- **FR-002**: When cloud mode is NOT selected, the system MUST behave byte-for-byte as it does today — same local Qwen feedback flow, same report output and `schema_version`, no OpenRouter token prompt, and no network calls beyond the existing model-download path.
- **FR-003**: When cloud mode IS selected, the system MUST generate the grammar/coherence feedback via an OpenRouter-hosted model instead of the local Qwen model, and MUST NOT load the local Qwen weights into memory during that run.
- **FR-004**: When cloud mode is selected and no OpenRouter token is stored, the system MUST prompt the user once for their OpenRouter API token and then persist it in the project's existing per-user configuration location.
- **FR-005**: On subsequent cloud-mode runs with a stored token, the system MUST reuse the stored token silently and MUST NOT prompt again, unless the stored token is missing or rejected by OpenRouter.
- **FR-006**: When the OpenRouter token is missing (and cannot be obtained) or is rejected by OpenRouter, the system MUST surface a clear, actionable error that tells the user how to update the token AND how to fall back to local mode.
- **FR-007**: The system MUST expose the OpenRouter model id as a single `model:` key in a dedicated YAML config file at `~/.speakloop/openrouter.yaml`, editable with no code change. There is no environment-variable override for the model id — this file is the one designated place.
- **FR-008**: The OpenRouter model id MUST default to `qwen/qwen3.7-max` when `~/.speakloop/openrouter.yaml` is absent or its `model:` key is unset.
- **FR-009**: Changing only the `model:` value in `~/.speakloop/openrouter.yaml` and re-running the cloud command MUST cause the newly configured model to be used, with no code change required.
- **FR-010**: The cloud-model system prompt MUST live in its own dedicated prompt file that the user can edit directly to tune cloud behavior, separate from any prompt used by the local Qwen flow.
- **FR-011**: Editing the cloud system-prompt file MUST change the cloud model's behavior on the next cloud run, with no code change required.
- **FR-012**: Cloud mode MUST NOT read, reference, modify, or depend on the local Qwen flow's prompt; the local prompt file MUST be left untouched by this feature.
- **FR-013**: The system MUST limit cloud mode to OpenRouter only — no generic multi-provider abstraction, no usage dashboards, no billing UI, and no automatic model selection.
- **FR-014**: Cloud-mode failures contacting OpenRouter (network error, timeout, server error, unparseable response) MUST be surfaced clearly and MUST degrade gracefully without crashing the session, consistent with how local-mode feedback failures are handled today.
- **FR-015**: Token storage MUST use the project's existing per-user configuration conventions; the feature MUST NOT introduce a new secrets manager or bespoke storage scheme.
- **FR-016**: The feature MUST be additive — it MUST NOT modify, refactor, or replace the local Qwen flow beyond the minimum branch needed at the entry point to choose between local and cloud mode.
- **FR-017**: The system MUST provide a way for the user to update/replace a stored OpenRouter token (e.g., re-prompting after a rejection, or a documented manual edit of the stored credential), so a rotated or corrected token can take effect.
- **FR-018**: On using cloud mode, the system MUST disclose in English that attempt transcripts are sent to OpenRouter for analysis (while audio recordings and reports stay on the device): a one-time disclosure at first-run token capture plus a brief reminder on each cloud run. Invoking the cloud opt-in is itself the consent; the system MUST NOT require a separate blocking confirmation prompt.

### Key Entities *(include if data involved)*

- **Cloud-mode selector**: The opt-in signal that switches a practice run from local to cloud feedback. Default is local (off).
- **OpenRouter credential (API token)**: A per-user secret entered once and persisted in the project's existing user-config location; read on each cloud run; replaceable when missing or rejected.
- **Cloud model-id setting**: The `model:` key in the dedicated YAML config file `~/.speakloop/openrouter.yaml`; defaults to `qwen/qwen3.7-max` when absent.
- **Cloud system-prompt file**: A dedicated, user-editable text file holding the instructions sent to the cloud model; wholly separate from the local-flow prompt.
- **Feedback request/response**: The grammar/coherence analysis exchanged with OpenRouter, producing the same kind of feedback the local model produces, consumed by the existing report writer.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A user who has never opted into cloud mode observes zero behavioral change: for identical inputs, the debrief report content and `schema_version` are identical to the pre-feature output, no OpenRouter token is ever requested, and the run completes with no network access available.
- **SC-002**: On a machine that cannot load the local Qwen model, a user can complete a full practice session including grammar feedback in cloud mode, and the local Qwen weights are never loaded during that run.
- **SC-003**: The first cloud-mode run prompts for a token exactly once; a second cloud-mode run with the stored token prompts zero times and completes using the stored token.
- **SC-004**: Changing only the `model:` value in `~/.speakloop/openrouter.yaml` (no code edit) routes the next cloud run to the newly configured model; with the file or `model:` key absent, the run uses `qwen/qwen3.7-max`.
- **SC-005**: Editing only the cloud system-prompt file (no code edit) produces an observable change in the cloud model's output on the next run, while the local-mode output for the same inputs is unchanged.
- **SC-006**: A missing or invalid token produces an error message that names both remediation paths — how to update the token and how to fall back to local mode — in 100% of missing/invalid-token cases.
- **SC-007**: A cloud-side failure (network, timeout, server error, unparseable response) never crashes the session; the rest of the debrief is preserved, matching local mode's graceful-degradation behavior.

## Assumptions

- **Cloud mode replaces only the language-model feedback step.** The mental model "today `speakloop practice` runs local Qwen; in cloud mode the same intent runs against an OpenRouter model" plus the fact that OpenRouter serves text LLMs means speech synthesis and transcription continue to use their existing local models. Cloud mode is not a fully-cloud pipeline; it removes only the local Qwen RAM burden, which is the largest model.
- **Opt-in shape.** The cloud opt-in is assumed to be a `--cloud` flag on the existing `practice` command, matching the project's existing per-command boolean-flag convention (e.g., `--listen-only`, `--no-audio`). The final shape (flag vs. subcommand vs. parallel command) is a project-fit decision deferred to planning; the requirements above are written to hold for any of them.
- **Token storage location.** The OpenRouter token is stored under the project's established per-user config root (`~/.speakloop/`), consistent with how the project already places user files there; the exact filename/format is a planning detail. No new secrets manager is introduced.
- **Token precedence.** When both an environment-provided token and a stored token are present, a documented, deterministic precedence applies (assumed environment variable first, then the stored credential), mirroring the project's existing env-over-file precedence patterns.
- **Model-id configuration** is the `model:` key in a dedicated YAML config file `~/.speakloop/openrouter.yaml` (honoring the constitution's YAML-for-user-config mandate and persisting across shells; this is the project's first general settings-YAML file). The **cloud system prompt** is its own separate file under `~/.speakloop/`. Both are editable with no code change and read only in cloud mode (FR-007–FR-011). The token remains a separate plain secret file, not part of this YAML (Token storage assumption above).
- **Constitution offline guarantee applies to the default/local path.** The "no network calls after model download" guarantee (Constitution Principle II) is preserved for the default experience. Cloud mode is a deliberate, user-invoked exception that makes network calls only while explicitly selected; this intentional friction is expected to be justified in the plan's Complexity Tracking (as the `aria2` prereq was for feature 007).
- **Graceful degradation parity.** Cloud-mode feedback failures reuse the existing graceful-degradation behavior of the local feedback path (the session continues and the report records the failure) rather than introducing new crash paths.
- **Report contract is unchanged.** Cloud-generated feedback is rendered through the existing report writer, so on-disk report layout, naming, and `schema_version` (1) are unchanged; any new frontmatter, if needed, would be additive only.
- **Networking dependency.** Cloud mode requires a working network connection and a valid OpenRouter account/token; provisioning the account and any associated cost are the user's responsibility and out of scope.

## Out of Scope

- Any provider other than OpenRouter, or a generic multi-provider abstraction layer.
- Usage dashboards, billing UI, spend tracking, or automatic/auto-selected model routing.
- Cloud replacement of the speech-synthesis or transcription steps.
- Changes to the local Qwen flow beyond the minimal entry-point branch needed to select cloud mode.
- A new secrets manager, keychain integration, or encrypted credential store beyond the project's existing user-config conventions.
- Changes to the report schema or the offline guarantee of the default local path.
