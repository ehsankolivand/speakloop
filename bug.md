## High

- [x] **BUG-001 — Report frontmatter parse truncates at a `---` line inside a block scalar (silent data loss)**
  - Severity: High
  - Category: Logic
  - Location: src/speakloop/feedback/frontmatter.py:307 — `_frontmatter_dict`
  - Why it's a bug: `parts = text.split("---\n", 2)` splits on the `---\n` **substring**, expecting the second occurrence to be the closing frontmatter fence. But `dump()` serializes `question`/`ideal_answer` as YAML literal block scalars indented by two spaces, so a source line equal to `---` is written as `  ---\n`, which *contains* `---\n` as a substring. That indented occurrence is matched before the real closing fence, so `parts[1]` is cut off mid-frontmatter. `yaml.safe_load(parts[1])` still succeeds on the (valid) truncated YAML, so the corruption is completely silent.
  - Impact: Any report whose `question_text` or `ideal_answer` contains a standalone `---` line (a normal Markdown horizontal rule in a free-form reference answer) re-parses into a `Session` that silently drops `attempts`, `grammar_patterns`, `generated_by_phase`, and every key serialized after the offending block. Reproduced end-to-end: an `ideal_answer` of `"Intro line\n---\nAfter the rule"` round-trips to a Session with `generated_by_phase="B"` (was `"C"`), `len(attempts)==0` (was 1), and `ideal_answer=="Intro line"`. This corrupts every consumer that re-reads reports — `resume`, `rebuild`, `trends`, `debrief` — and breaks the documented `dump → parse → dump` idempotency invariant.
  - How to fix: Stop splitting on the bare `---\n` substring. Anchor the fences to whole lines — e.g. split only where a line is exactly `---` (`re.split(r"(?m)^---$", text)` or a line-by-line scan), so indented block-scalar content containing `---` is never mistaken for the closing fence.
  - Confidence: Confirmed — traced `_frontmatter_dict` + `dump`, and reproduced the data loss with a real `Session` round-trip.
  - Resolution: Added a module-level `_FENCE_RE = re.compile(r"(?m)^---$")` and replaced the `text.split("---\n", 2)` substring split with `_FENCE_RE.split(text, maxsplit=2)` in `_frontmatter_dict` (src/speakloop/feedback/frontmatter.py). Whole-line-anchored fences never match an indented block-scalar `  ---`, so the closing fence is found correctly. commit 72d0bd2 (`fix(BUG-001): anchor frontmatter fence split to whole lines`); verified by reproducing the original truncation, confirming the fixed round-trip restores `generated_by_phase`/`attempts`/`ideal_answer` and stays idempotent, that a `---` inside `question_text` is also preserved, that normal reports and edge inputs (bare yaml, opening-fence-only, empty) are unchanged, plus the full suite (855 passed, 3 skipped) and an independent fresh-context investigator agent.

## Medium

- [x] **BUG-002 — `store.io.load` crashes on valid-JSON-but-shape-corrupt store instead of returning an empty Store**
  - Severity: Medium
  - Category: Functional
  - Location: src/speakloop/store/io.py:29 — `load` (root cause in `store/model.py:74,77,81` — `Store.from_dict`)
  - Why it's a bug: `load()` wraps only `json.loads` in its `try/except (json.JSONDecodeError, OSError, UnicodeDecodeError)` (io.py:25-28); the `Store.from_dict(data)` call on line 29 is unguarded. Inside `from_dict`, `schedule_raw = d.get("schedule") or {}` (model.py:74) accepts a truthy non-dict, then `schedule_raw.items()` (model.py:77) raises `AttributeError` on a list; and `int(d.get("store_version", STORE_VERSION))` (model.py:81) raises `ValueError`/`TypeError` on a non-numeric/`null` value. These escape `load()`.
  - Impact: A JSON-valid but wrong-shape `~/.speakloop/store.json` (external tampering or a hand-edit) makes `load()` raise instead of returning an empty, rebuildable `Store` — directly contradicting the module docstring ("missing / corrupt / older-version file loads as an empty Store") and `store/CLAUDE.md`. Every command that loads the store (`today`, `practice`, `resume`, `rebuild`) then crashes with a raw traceback, and recovery requires the user to manually delete the file — the exact outcome the cache contract exists to prevent. Reproduced: `{"schedule": [1,2,3]}` → `AttributeError`; `{"store_version": "abc"}` → `ValueError`; `{"store_version": null}` → `TypeError`.
  - How to fix: Wrap `Store.from_dict(data)` (io.py:29) in `try/except (ValueError, TypeError, AttributeError, KeyError): return Store()`. Additionally harden `from_dict`: guard `int()` around `store_version`, and require `isinstance(schedule_raw, dict)` before iterating (mirroring the existing per-entry `isinstance(e, dict)` guard).
  - Confidence: Confirmed — traced `load` + `from_dict` and reproduced all three crash inputs.
  - Resolution: Wrapped the `Store.from_dict(data)` call in `store.io.load` in `try/except (ValueError, TypeError, AttributeError, KeyError): return Store()` — the single contract-owning site, mirroring the existing `json.loads` guard right above it (src/speakloop/store/io.py). Added a parametrized regression test (`test_load_valid_json_but_shape_corrupt_returns_empty`) covering `{"schedule":[1,2,3]}`, `{"store_version":"abc"}`, `{"store_version":null}` (tests/unit/store/test_rebuild.py). commit 01f407a; verified by reproducing all three original crashes now returning an empty Store, confirming a well-shaped store still loads its data unchanged, the store suite (14 passed) and the store-consumer suite (today/resume/pronounce/coordinator/coverage — 41 passed), plus an independent fresh-context investigator agent.

- [x] **BUG-003 — Unhashable `difficulty` / `type` value crashes `content.schema.parse` with an uncaught TypeError**
  - Severity: Medium
  - Category: Functional
  - Location: src/speakloop/content/schema.py:121 and :136 — `parse`
  - Why it's a bug: Both `difficulty not in _VALID_DIFFICULTIES` (line 121) and `qtype not in _VALID_TYPES` (line 136) perform a set-membership test on the raw YAML value with no type guard. When the value is a YAML list or mapping (e.g. `difficulty: [easy, hard]` or `type: [behavioral]`), it is unhashable and Python raises `TypeError: unhashable type: 'list'`. Every other malformed-value path in `parse()` degrades to a warning or a `QASchemaError`; the neighboring `tags` field (line 126) even has an explicit `isinstance(tags, list)` guard — showing graceful degradation was the intent here too.
  - Impact: A hand-edited Q&A file with a list/dict `difficulty` or `type` value makes `content.loader.load()` propagate a raw uncaught `TypeError` (verified: `loader.load` wraps `parse` in `except QASchemaError` only, loader.py:42-44) instead of the intended graceful warning + fallback (`difficulty=None` / `type="definition"`) or a clean `QALoadError` naming the offending entry. The user sees a traceback instead of an actionable message — a broken authoring experience for the documented "bring your own questions" flow.
  - How to fix: Only run each membership test when `isinstance(value, str)`; for any non-string value, append the existing "unknown …; ignored/treated as definition" warning and set the field to its default — mirroring the `tags` handling on lines 126-128. Apply the guard at both line 121 and line 136.
  - Confidence: Confirmed — reproduced `parse({'schema_version':1,'questions':[{'id':'q1','question':'x','ideal_answer':'y','difficulty':['easy','hard']}]})` raising `TypeError`, escaping `loader.load`'s `except QASchemaError`.
  - Resolution: Added an `isinstance(..., str)` short-circuit before each set-membership test in `content.schema.parse` — `difficulty` (line ~121) and `type` (line ~136) — so a non-string (list/dict) value degrades to the existing "unknown …; ignored/treated as definition" warning + default instead of raising `TypeError` (src/speakloop/content/schema.py). Added parametrized regression tests for both fields (tests/unit/content/test_type_field.py). commit 79f01ee; verified by reproducing the four original crashes (list/dict × difficulty/type) now degrading gracefully, confirming valid values (and unknown strings) parse byte-identically with no spurious warnings, an end-to-end `loader.load` on a bad file returning a QAFile with warnings (no traceback), the content + questions-CLI suites (31 passed) and the question-loading consumer suite (59 passed), plus an independent fresh-context investigator agent.

- [ ] **BUG-004 — Retry scoring error is mislabeled "still off": contradictory live output + false verdict persisted to the report**
  - Severity: Medium
  - Category: Logic
  - Location: src/speakloop/pronunciation/drill_runner.py:437-458 — `run_drill_item`
  - Why it's a bug: When the bounded automatic retry pass returns status `"error"` (the wav2vec2 scorer raises, or the mic/recorder raises on the second read), the code prints the actionable error via `_print_outcome(console, "error", [], r_detail)` (line 439) but then sets `outcome = "still_off"` (line 440). The retry vocabulary has no error state, so a scoring failure is conflated with a scored-but-still-flagged result. After the loop, `outcome == "still_off"` falls through to the `else` branch (lines 456-457) and prints "Still a little off — keep practising; moving on.", and `item["retry"]["outcome"] = "still_off"` (line 458) is persisted. The comment on line 438 explicitly says the intent is to avoid "pretending it was 'still a bit off'" — but the code does exactly that.
  - Impact: On any retry-attempt error the learner sees two contradictory live lines — "I couldn't score that one — [mic/model hint]" immediately followed by "Still a little off — keep practising" — and the written report's Pronunciation section renders `still_off` (via `feedback._retry_line`, feedback.py:36-37) as "On retry: still a little off — worth more practice." The report thus asserts a false pronunciation verdict for a retry that was never actually scored, telling the learner to practice a sound that may be fine.
  - How to fix: Add a distinct retry outcome for the error case (e.g. `"error"`/`"could_not_score"`) set at line 440; handle it in the post-loop print branch (lines 452-457) so it does not print "Still a little off"; and add a matching branch in `feedback._retry_line` (or return `None`) so the report does not claim the sound is "still off". Do not reuse `"still_off"` for a scoring/mic failure.
  - Confidence: Confirmed — traced the retry loop, the post-loop print branch, and `feedback._retry_line`; the "error → still_off" assignment and its persisted rendering are all present in code.

## Low

- [ ] **BUG-005 — Coverage state membership check is case-sensitive, silently downgrading a capitalized `Covered`/`Partial` to `missed`**
  - Severity: Low
  - Category: Logic
  - Location: src/speakloop/coverage/scoring.py:50-51 — `_coverage_records`
  - Why it's a bug: `state = str(c.get("state", "")).strip()` (line 50) is never lowercased, while `_VALID_STATES` is lowercase-only, so line 51's `state if state in _VALID_STATES else "missed"` coerces any capitalized-but-valid state (e.g. `"Covered"`, `"Partial"`) to `"missed"`. `_aggregate` (lines 34-36) then counts against lowercase `"covered"`/`"partial"`, so the point is dropped from the score. Sibling LLM-output handlers in the same feature normalize case before comparing (`content_errors.py:21`, `triage/mishearing.py`, `warmup/drill.py:75`), so this is a genuine deviation from the established pattern.
  - Impact: If the coverage LLM returns a capitalized state, that key point is scored `missed` instead of covered/partial, lowering the attempt's coverage aggregate below what the model assigned and producing a wrong answer-quality grade (which drives SRS scheduling). Bounded to that attempt; no crash.
  - How to fix: Lowercase before the membership test — `state = str(c.get("state", "")).strip().lower()` — keeping the `state in _VALID_STATES else "missed"` check unchanged.
  - Confidence: Suspected — the code path and downgrade are confirmed by reading, but the trigger requires the model to deviate from the coverage prompt's explicit lowercase-JSON example (temperature 0.2); I could not confirm that capitalized states actually occur in practice.
