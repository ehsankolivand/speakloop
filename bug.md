## Low

- [x] **BUG-001 — A stale `*.aria2` / `*.incomplete` sidecar makes a fully-downloaded model validate as `incomplete` forever (cross-backend), causing a permanent install failure**
  - Severity: Low
  - Category: Logic
  - Origin: Regression (from IMP-002, commit `c66e1b7`)
  - Location: src/speakloop/installer/validator.py:42-44 & :59-65 — `_has_incomplete_download` / `validate`; root cause is the absence of sidecar cleanup in `src/speakloop/installer/downloader.py` (`download_model` / the `snapshot_download` fallback at :306-311)
  - What the bug is: IMP-002 added a pre-size-test scan (`_has_incomplete_download`) that returns `reason="incomplete"` whenever any `*.aria2` control file or `*.incomplete` marker remains anywhere under `model.local_path`. But nothing in the installer ever removes those markers — the only `.unlink()` calls (`downloader.py:276,285`) delete metadata files, and a `*.aria2` is removed only by aria2 itself on its *own* successful completion. So if a download is interrupted under one backend (leaving that backend's marker next to already-complete bytes) and then finished under the *other* backend — aria2 hard-killed then completed by the `snapshot_download` fallback (aria2 uninstalled between runs), or a snapshot-left `.incomplete` under `.cache/huggingface/download/` then completed by a now-installed aria2 — the first backend's marker is never cleaned up, so `validate()` reports `incomplete` on every subsequent run even though the bytes are complete.
  - Why it's a bug: `_missing_or_invalid` (`installer/__init__.py:53`) treats `incomplete` as invalid, so `_ensure` re-downloads and re-validates (`__init__.py:82-91`). The completing backend never removes the *other* backend's marker, so `still_bad` stays non-empty and `_ensure` raises `InstallFailedError("Validation failed after download: …")` on **every** run — a permanent hard failure whose message never names the hidden sidecar, so the user has no way to know the fix is to delete one file. Before IMP-002 the same directory validated OK (its byte sum clears the ±25% tolerance), so this is a new failure mode the change introduced. The IMP-002 resolution note's own premise — that a not-ok result "re-queues it so aria2 `--continue` / snapshot `resume_download` finish it" — holds for a *same-backend* resume but is false for the cross-backend case, because neither backend removes the other's marker.
  - How to fix it: After a successful `download_fn`, remove orphaned `*.aria2`/`*.incomplete` sidecars whose target file exists and clears the size check (covering both backends); or make `_has_incomplete_download` size-aware so a sidecar only blocks an under-sized directory (a complete directory validates OK regardless of a stale marker); at minimum, name the offending sidecar path in the raised `InstallFailedError` so the failure is self-diagnosing. Add a validator/installer test that drops a stale `<shard>.aria2` next to an otherwise full-size directory and asserts the provisioning path resolves rather than looping to a hard failure.
  - Confidence: Suspected — I confirmed directly in code that (a) no code path removes `*.aria2`/`*.incomplete` sidecars (only metadata files are unlinked), (b) the `snapshot_download` fallback writes to the same `model.local_path` that `validate` scans, so "complete bytes + leftover sidecar" is a real state, and (c) a repeated `incomplete` verdict drives `_ensure` to raise `InstallFailedError` permanently. What I could NOT verify — no ability to execute `aria2c`/`snapshot_download` in this read-only pass — is that the triggering runtime state is actually reached: specifically that aria2 retains a `<shard>.aria2` control file after a SIGKILL with complete bytes, and that a real user hits the cross-backend switch (aria2 present→absent, or absent→installed) between the interrupted attempt and the retry. A same-backend resume self-heals (aria2 `--continue` removes its own control file; `snapshot_download` renames its own `.incomplete`), so this is a narrow, recoverable, non-default-path failure rather than a common-path crash — hence Low.
  - Resolution: Confirmed real on re-investigation (the three load-bearing facts trace out in code; the runtime trigger stays hard to reach but the code-level gap is definite). Added `validator.clear_incomplete_markers(path)` (removes `*.aria2`/`*.incomplete` markers under a path, reusing `_INCOMPLETE_GLOBS` so detect and clear share one glob source) and call it at the END of `downloader.download_model`, after either backend RETURNS success. Success is the authoritative completion signal: a genuinely-interrupted download raises (aria2 `HARD_FAILURE`/exhausted-transient, `snapshot_download` exception) and never reaches the cleanup, so IMP-002's guarantee is preserved (a killed-at-~80% download still validates `incomplete`). The cleanup removes the CROSS-BACKEND leftover the completing backend cannot remove, so `_ensure`'s re-download → re-validate now passes instead of looping to a permanent `InstallFailedError`. Files: `src/speakloop/installer/validator.py`, `src/speakloop/installer/downloader.py`, `src/speakloop/installer/CLAUDE.md`. Landed as commit `fix(BUG-001): clear stale cross-backend download markers on success`; verified by 2 new validator tests (incomplete→ok round-trip; no-op when clean) + 1 downloader integration test (snapshot completes while an orphaned `.aria2` remains → cleared, no longer `incomplete`), the installer suite + help-isolation + context-budget gates (35 passed), the full suite (**926 passed**, +3), `ruff`/`mypy` clean on the touched files, and an independent fresh-context reviewer (verdict: SOUND — loop broken, IMP-002 preserved, no import cycle/engine leak, control flow unchanged).
